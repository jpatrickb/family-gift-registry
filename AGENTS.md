# Agent instructions

## Before writing any code

Read [`CODEBASE.md`](./CODEBASE.md). It describes the current project structure, design decisions, and known patterns. Do not assume anything about the codebase that isn't confirmed by reading the actual files.

## After making changes

Update `CODEBASE.md` to reflect what changed. Keep it accurate and current — it's the single source of truth for future agents. If you add a route, component, or table, document it. If you change a design decision, update the relevant section.

## Key conventions

- **Server vs client components**: default to Server Components. Use `"use client"` only when you need browser events, state, or effects.
- **Supabase clients**: use `createClient()` from `src/lib/supabase/server.ts` in Server Components and Route Handlers; `getSupabaseBrowserClient()` from `src/lib/supabase/client.ts` in Client Components; `createAdminClient()` from `src/lib/supabase/admin.ts` only in Route Handlers that must bypass RLS.
- **TypeScript types**: `src/types/database.types.ts` is generated by `npx supabase gen types typescript`. Do not hand-edit it. Run the command again after any schema changes.
- **Surprise preservation**: the `gift_claims` RLS SELECT policy is the core security guarantee — gift owners cannot see claims on their own gifts. Never weaken this policy. Any code that reads claim data must respect this; the DB enforces it as a last line of defense.
- **Forms**: use `react-hook-form` + `zod` with schemas defined in `src/lib/validations.ts`.
- **Toasts**: use `sonner` via `import { toast } from "sonner"`.
